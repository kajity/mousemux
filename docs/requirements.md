# mousefold 要件定義書

## 1. 目的

本システムは、Linux 上で物理マウスから入力されるイベントを監視し、定義済みのルールに従ってイベントを仮想マウスおよび仮想キーボードへ振り分ける常駐デーモンである。

選択した物理マウスは `grab` して入力を独占し、未リマップの移動・ホイール・通常ボタンは仮想マウスへ流し、リマップ対象だけをキーボード入力へ変換する。systemd によって起動・停止・監視されることを前提とする。

## 2. スコープ

### 2.1 対象

- 物理マウスの単一デバイス監視
- 対象物理マウスの `grab`
- マウスボタン入力を契機とするキーボード入力へのリマップ
- 未リマップの移動・ホイール・通常ボタンの仮想マウスへのパススルー
- 設定ファイルによるリマップルール定義
- systemd 管理下での常駐実行
- Linux の `evdev` / `uinput` を利用した入力監視と仮想デバイス生成

### 2.2 非対象

- 複数マウス同時監視
- 移動イベントやホイールイベントのキーボード変換
- GUI 設定ツール
- 動的なクラウド連携
- Wayland/X11 固有 API に依存した実装
- 高度なマクロ編集 UI

## 3. 想定利用環境

- OS: systemd を採用する Linux ディストリビューション
- 入力取得: `/dev/input/event*`
- 仮想デバイス生成: `/dev/uinput`
- 実行形態: systemd service として常駐
- 実装言語: Rust

## 4. 用語定義

- 物理入力デバイス: 実機のマウス
- 仮想マウス: `uinput` により生成されるマウスデバイス
- 仮想キーボード: `uinput` により生成されるキーボードデバイス
- リマップルール: 物理マウス入力を仮想キーボード入力へ変換する定義
- パススルー: 物理マウスの入力を仮想マウスへそのまま流すこと
- 設定ファイル: YAML または独自文法で記述されるルールファイル

## 5. システム概要

本システムは以下の処理を行う。

1. 対象マウスデバイスを特定する
2. 設定ファイルを読み込み、リマップルールを構築する
3. 対象デバイスを `grab` する
4. `uinput` により仮想マウスと仮想キーボードを生成する
5. マウスイベントを監視する
6. 一致するリマップルールが存在するボタンイベントは仮想キーボードへ送出する
7. それ以外の移動・ホイール・未リマップボタンは仮想マウスへ送出する
8. systemd により異常終了時の再起動とログ収集を行う

## 6. 機能要件

### 6.1 デバイス監視

- FR-001: システムは 1 台の物理マウス入力デバイスを監視できること
- FR-002: 監視対象デバイスはデバイスパス、by-id パス、または識別子ベースで指定できること
- FR-003: 指定デバイスが存在しない場合、明確なエラーログを出力して終了または再試行できること
- FR-004: 同時に複数マウスを監視対象として扱わないこと
- FR-005: 指定した物理マウスを `grab` できること
- FR-006: `grab` に失敗した場合、明確なエラーログを出力して終了できること

### 6.2 リマップ処理

- FR-010: システムはマウス入力を契機としてキーボードイベントへ変換できること
- FR-011: 少なくとも押下と解放の両イベントを扱えること
- FR-012: 1 つのマウス入力に対して 1 つ以上のキーボードイベント列を出力できること
- FR-013: 初期実装でキーボード変換対象とする入力はマウスボタンイベントに限定すること
- FR-014: 将来的な拡張を考慮し、単発キー送出以外に複数キー送出へ対応可能な内部構造とすること
- FR-015: 同一入力条件に対して複数ルールが競合する場合、設定ファイル内で後に記述されたルールを優先すること
- FR-016: 競合ルールを検出した場合、起動時または設定再読込時に警告ログを出力すること
- FR-017: リマップ対象イベントは仮想マウスへ送出しないこと

### 6.3 パススルー処理

- FR-020: 未リマップのマウスボタンイベントは仮想マウスへ送出できること
- FR-021: 相対移動イベントは仮想マウスへ送出できること
- FR-022: ホイールイベントは仮想マウスへ送出できること
- FR-023: パススルー対象イベントは、元の入力順序を極力保って仮想マウスへ送出すること
- FR-024: リマップ対象とパススルー対象を二重送出しないこと

### 6.4 設定ファイル

- FR-030: リマップ定義は外部設定ファイルから読み込めること
- FR-031: 設定フォーマットは YAML または独自文法のいずれかを採用できること
- FR-032: 初期実装では YAML を優先し、独自文法は将来拡張としてもよい
- FR-033: 設定ファイルの構文エラー時は、行番号または該当ルールを含む診断情報を出力すること
- FR-034: 設定ファイルには少なくとも以下を記述できること
- FR-035: 対象デバイスの指定
- FR-036: 入力条件
- FR-037: 出力するキーコード
- FR-038: 押下時と解放時の動作
- FR-039: 設定ファイルパスはコマンドライン引数または固定パスで指定できること
- FR-040: 設定ファイルの変更を検知し、プロセス再起動なしでルールを再読込できること
- FR-041: 再読込失敗時は直前に有効だった設定を保持し、エラーログを出力すること

### 6.5 仮想デバイス

- FR-050: システムは仮想マウスを生成すること
- FR-051: システムは仮想キーボードを生成すること
- FR-052: 仮想マウスは OS からマウスとして認識されること
- FR-053: 仮想キーボードは OS からキーボードとして認識されること
- FR-054: 仮想マウスには移動、ホイール、未リマップボタンの送出に必要な capability を登録できること
- FR-055: 仮想キーボードには設定で使用される全キーコードを登録できること

### 6.6 デーモン運用

- FR-060: systemd service として起動可能であること
- FR-061: フォアグラウンド実行を前提とし、systemd にプロセス管理を委譲すること
- FR-062: systemd による自動再起動設定を可能とすること
- FR-063: 標準出力または標準エラーへログを出力し、journald で収集可能であること
- FR-064: 停止シグナルを受けた場合は安全に終了処理を行うこと
- FR-065: systemd service は root 権限で実行すること

## 7. 設定仕様要件

### 7.1 YAML 採用時の最低要件

- NFR-100: 人手で編集しやすいこと
- NFR-101: Rust の `serde` 系ライブラリで安定してパース可能であること
- NFR-102: コメント記述または可読性を確保できること

### 7.2 独自文法採用時の最低要件

- NFR-110: YAML より高い表現力または簡潔性があること
- NFR-111: パーサ実装コストと保守コストを明確化できること
- NFR-112: 構文エラー時の診断品質を担保できること

### 7.3 推奨方針

- 初期リリースは YAML を採用する
- 独自文法は、複雑なマクロ、条件分岐、レイヤー切替などが必要になった場合に再検討する

## 8. 非機能要件

### 8.1 性能

- NFR-001: 入力イベント検知から仮想マウスまたは仮想キーボード送出までの遅延は、通常利用で体感上問題ない水準であること
- NFR-002: 常駐時の CPU 使用率は低く抑えられること
- NFR-003: 不要なポーリングを避け、イベント駆動で処理すること

### 8.2 信頼性

- NFR-010: 設定不備、デバイス未接続、権限不足、grab 失敗を識別可能な形でログ出力すること
- NFR-011: 異常終了時に systemd の再起動ポリシーと整合すること
- NFR-012: 仮想デバイス生成失敗時は中途半端な状態を残さず終了すること
- NFR-013: ホットリロード中に一時的な設定更新失敗が発生しても、入力処理を継続できること
- NFR-014: 物理デバイスの直接入力と仮想デバイスからの入力による二重入力を避けること

### 8.3 保守性

- NFR-020: 入力取得、設定パース、ルーティング判定、仮想デバイス出力をモジュール分離できる構造とすること
- NFR-021: ルール追加時に既存処理への影響を局所化できること
- NFR-022: ログメッセージは障害解析に必要な識別情報を含むこと

### 8.4 セキュリティ

- NFR-030: root 実行を前提としつつ、systemd の保護設定により不要な権限拡大を抑制すること
- NFR-031: 設定ファイルは信頼できる管理者のみが編集可能であること
- NFR-032: 任意コード実行につながる設定解釈は行わないこと

## 9. systemd 要件

### 9.1 サービス構成

- systemd unit は `Type=simple` を基本とする
- デーモン自身はバックグラウンド化しない
- `Restart=on-failure` を推奨する
- 設定ファイルパスや対象デバイス指定は `ExecStart` 引数または `EnvironmentFile` で渡せる設計とする
- `User=root` または root 実行相当の systemd 設定を前提とする
- 設定変更のホットリロードはプロセス内監視を基本とし、systemd の再起動に依存しない

### 9.2 運用要件

- サービス起動時に設定ファイル妥当性とデバイスアクセス可否を検証すること
- 対象デバイスを `grab` できることを起動時に確認すること
- journald から起動失敗原因を追跡可能であること
- OS 起動時の自動起動を可能にすること
- 競合ルール検出時は起動継続しつつ警告ログを残すこと

## 10. 制約条件

- Linux の `evdev` と `uinput` が利用可能であること
- 実行ユーザーは root であること
- キー送出は Linux 入力サブシステムが扱えるキーコードに限定されること
- 初期実装でキーボード変換対象とする入力はボタンイベントのみであること

## 11. 想定設定例

```yaml
device:
  path: /dev/input/by-id/usb-Logitech_Gaming_Mouse-event-mouse

remaps:
  - input:
      type: key
      code: BTN_RIGHT
      value: 1
    output:
      - key: KEY_LEFTMETA
        value: 1

  - input:
      type: key
      code: BTN_RIGHT
      value: 0
    output:
      - key: KEY_LEFTMETA
        value: 0
```

## 12. 受け入れ条件

- AC-001: 指定した物理マウス入力を監視できること
- AC-002: 指定した物理マウスを `grab` できること
- AC-003: 設定ファイルのルールに従い、リマップ対象を仮想キーボードからキー送出できること
- AC-004: 未リマップの移動・ホイール・通常ボタンを仮想マウスへ送出できること
- AC-005: OS から仮想マウスがマウスとして見えること
- AC-006: OS から仮想キーボードがキーボードとして見えること
- AC-007: systemd service として起動、停止、再起動できること
- AC-008: 設定不正時に起動失敗と理由がログで確認できること
- AC-009: 設定ファイル変更時にプロセス再起動なしで新ルールへ切り替わること
- AC-010: 無効な再読込が発生しても既存ルールで動作継続すること
- AC-011: 競合ルールがある場合、後勝ちで解決され、かつ警告ログが残ること
- AC-012: 二重入力が発生しないこと

## 13. 実装方針の推奨

- 第 1 段階: YAML のみ対応して最小実装を成立させる
- 第 2 段階: grab、仮想マウス、仮想キーボード、基本ルーティングを実装する
- 第 3 段階: 複数キー出力、設定検証強化、ホットリロードを行う
- 第 4 段階: 必要性が明確になった場合のみ独自文法を追加する

## 14. 決定事項

- 監視対象は単一マウスとする
- 監視対象デバイスは `grab` する
- 初期実装でキーボード変換する対象はボタンイベントのみとする
- 未リマップの移動・ホイール・通常ボタンは仮想マウスへ流す
- 設定変更のホットリロードを実装する
- systemd service は root 実行とする
- 競合ルールは後勝ちで解決し、同時に警告ログを出力する

- 移動イベントのキーボード変換
