# remapper-test-diagnostics

## 目的
この skill は、実装したデーモンが要件を満たしているかを確かめるために使う。
単なる unit test だけでなく、手動検証、journald での診断、受け入れ条件へのトレースを含む。

## この skill を使う場面
- どのテストを書くべきか整理したい
- 不具合の切り分けをしたい
- ホットリロード失敗時の挙動を確認したい
- systemd 配下での挙動を確認したい
- grab とパススルーが正しく動くか確認したい

## 検証の基本方針
テストは 3 層に分ける。

### 1. Unit tests
ロジック単体を検証する。

対象:
- YAML パース
- 論理検証
- 競合解決
- ルーティング判定
- 未定義入力のパススルー判定

### 2. Integration tests
モジュール連携を検証する。

対象:
- 設定読込からルール構築まで
- ホットリロード成功
- ホットリロード失敗時のロールバック
- 仮想マウス出力
- 仮想キーボード出力
- フォアグラウンド実行

### 3. Manual tests
実機依存の最終確認。

対象:
- 対象マウスだけ反応する
- 物理デバイスが `grab` される
- リマップ対象ボタンで期待キーが出る
- 未リマップボタンが仮想マウス経由で機能する
- 移動とホイールが仮想マウス経由で機能する
- 設定変更が再起動なしで反映される
- 無効設定でも旧設定で動き続ける

## 受け入れ条件トレース
各変更で、最低限次を説明できること。

- AC-001: 指定物理マウスを監視できるか
- AC-002: 指定物理マウスを `grab` できるか
- AC-003: リマップ対象をキーボードへ変換できるか
- AC-004: 未リマップの移動・ホイール・ボタンを仮想マウスへ流せるか
- AC-005: systemd で起動・停止・再起動できるか
- AC-006: 設定不正時に理由がログで見えるか
- AC-007: ホットリロードできるか
- AC-008: 無効再読込でも既存動作継続か
- AC-009: 競合が後勝ち＋警告ログか
- AC-010: 二重入力にならないか

## 典型的な不具合と切り分け
### 1. 反応しない
確認順:
- 対象デバイスパスが正しいか
- root 権限か
- `grab` に成功しているか
- 入力イベントが読めているか
- ルールが一致しているか
- 仮想マウス/キーボードが作れているか

### 2. 二重入力になる
確認順:
- 物理デバイスが本当に `grab` されているか
- リマップ対象を仮想マウスへも送っていないか
- 同じイベントを二重 emit していないか

### 3. 押しっぱなしになる
確認順:
- 解放ルールがあるか
- 解放イベントを落としていないか
- 出力側で解放を送っているか

### 4. 設定変更が反映されない
確認順:
- watcher が反応しているか
- debounce が長すぎないか
- 新設定の検証に失敗していないか
- 旧設定維持ログが出ていないか

## 推奨テスト名
```text
tests/config_parse.rs
tests/config_validate.rs
tests/conflict_resolution.rs
tests/router_resolve.rs
tests/passthrough_behavior.rs
tests/reload_success.rs
tests/reload_rollback.rs
```

## 推奨コマンド
```bash
cargo fmt --all
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all
RUST_LOG=debug cargo run -- --config ./examples/config.yaml
journalctl -u ex-g-pro-remapper.service -b
```

## レビュー時の観点
- テストが要件に結びついているか
- Linux 依存部分をモック可能な境界に閉じ込めているか
- ログが診断可能な粒度か
- 失敗時に旧設定維持になっているか
- grab とパススルーがコード上で明確か

## 出力の期待形
この skill を使ったときは、次を返す。

- 追加/更新すべきテスト一覧
- 各テストの目的
- 手動確認手順
- 想定ログ
- 受け入れ条件との対応表
