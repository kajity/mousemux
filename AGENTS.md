# AGENTS.md

## Project
mousemux

## Mission
Linux 上で動く Rust 製の常駐デーモンを実装する。
このデーモンは、単一の物理マウスを `grab` して入力を独占し、YAML 設定に従ってイベントを 2 系統へ振り分ける。

- リマップ対象イベントは仮想キーボードからキーイベントとして送出する
- 未リマップの移動・ホイール・通常ボタンは仮想マウスへそのまま流す

## Product constraints
このプロジェクトでは以下を厳守すること。

- 入力対象は単一デバイス
- 選択した物理マウスは `grab` して、元デバイスからのイベントを OS へ直接流さない
- 初期実装でキーボードへ変換する対象はマウスボタンイベントのみ
- 移動イベントとホイールイベントは初期実装ではキーボード変換しない
- 未リマップの移動・ホイール・ボタンは仮想マウスへパススルーする
- リマップ対象イベントは仮想マウスへ流さず、仮想キーボードへだけ送る
- 設定フォーマットは初期実装では YAML を採用する
- systemd 管理下でフォアグラウンド実行する
- root 実行を前提とする
- 設定変更のホットリロードを実装する
- ホットリロード失敗時は旧設定を維持する
- 競合ルールは後勝ちで解決し、警告ログを出す
- Wayland/X11 固有 API に依存しない
- GUI ツールは作らない
- 任意コード実行につながる設定解釈は行わない

## Architecture policy
実装は以下の責務分離を基本とする。

- `config`
  - YAML 読み込み
  - 構文検証
  - 論理検証
  - 競合ルール検出
- `device`
  - 物理マウスのオープン
  - `grab` / `ungrab`
  - `evdev` イベント受信
  - 元デバイス capability の取得
- `virtual_mouse`
  - `uinput` 仮想マウス生成
  - 未リマップイベントのパススルー送出
- `virtual_keyboard`
  - `uinput` 仮想キーボード生成
  - キー送出
- `router`
  - 入力イベントとルールの照合
  - パススルーかキーボード変換かを判定
- `reload`
  - 設定ファイル変更監視
  - 新設定の読込と差し替え
  - 失敗時ロールバック

## Shared-state policy
共有状態は最小化する。
必要な共有状態は以下に限定する。

- 現在有効な設定
- 現在有効なコンパイル済みルール集合
- 仮想マウスハンドル
- 仮想キーボードハンドル
- 現在監視中の物理デバイス情報

`Arc<RwLock<...>>` または同等の読多書少構造を基本とし、ロック範囲は短く保つこと。

## Runtime policy
単一プロセス内で以下の責務を処理する。

1. 入力イベント受信タスク
2. 設定ファイル監視タスク
3. 終了シグナル監視タスク

複雑な並列化より、保守しやすい逐次処理または単純なタスク分離を優先する。

## Event routing policy
イベント処理では次を守ること。

- 物理マウスから受信したイベントを正規化する
- リマップ対象のボタンイベントなら仮想キーボードへ送る
- 非対象イベントなら仮想マウスへそのまま流す
- `SYN_REPORT` などのフレーム境界は出力先の整合性を保つ
- キー変換とマウスパススルーを二重送出しない
- 実行時の複雑さを減らすため、競合解決はロード時に済ませる

## Implementation stages
実装は必ず以下の順序で進めること。

### Stage 1
- YAML 読込
- 設定バリデーション
- エラー診断の整備

### Stage 2
- 単一マウス入力のオープンと `grab`
- 元デバイス capability の取得
- 仮想マウス生成
- 仮想キーボード生成

### Stage 3
- 1:1 リマップ
- 未リマップイベントの仮想マウスへのパススルー
- 複数キー出力

### Stage 4
- 競合検出
- 警告ログ
- ホットリロード
- ロールバック
- 再生成が必要なリソースの安全な差し替え

### Stage 5
- systemd unit
- 運用確認
- 受け入れ条件に沿った手動試験と自動試験

## Coding rules
- 変更前に、目的・影響範囲・失敗モードを短く整理する
- 1 回の変更で 1 つの責務に集中する
- public API には意図が分かる rustdoc を付ける
- `unwrap()` と `expect()` は本番コードで原則禁止
- エラー型は文脈を保持する
- ログは `ERROR/WARN/INFO/DEBUG` の用途を守る
- Linux 依存部分はモジュール境界の内側へ閉じ込める
- 設定ファイルのバリデーションはパース後に必ず論理検証も行う
- 実行時の分岐を減らすため、競合解決はロード時に済ませる
- 将来拡張を見越しても、今は過剰抽象化しない
- 要件にない機能を勝手に広げない

## Config rules
設定は以下を表現できる必要がある。

- 対象デバイス指定
- 入力条件
- 出力キー列
- 押下時と解放時の動作
- リロード設定

設定エラー時は、できるだけ以下を出すこと。

- 失敗した項目
- 原因
- 該当ルール
- 行番号相当の情報
- 修正の方向性

## Conflict rules
競合とは「同一入力条件に対して複数ルールが存在する状態」とする。
競合時は以下の動作を実装すること。

- 後に書かれたルールを採用
- 先のルールを無効化扱い
- 起動時または再読込時に `WARN` を出す
- ログには入力条件と競合したルール位置を含める

## Hot reload rules
ホットリロードでは以下を守ること。

- ファイル変更検知後に再読込する
- 新設定は構文検証と論理検証の両方に通ってから有効化する
- 失敗時は旧設定を維持する
- 入力変換処理は継続する
- 物理デバイス変更や capability 変更があるなら安全に再構築する
- 一時ファイル保存や editor save による多重イベントを考慮して debounce する

## Logging policy
ログレベルの使い分けは固定する。

- `ERROR`
  - 起動失敗
  - 設定読込失敗
  - デバイスオープン失敗
  - `grab` 失敗
  - uinput 生成失敗
- `WARN`
  - 競合ルール
  - ホットリロード失敗時の旧設定維持
- `INFO`
  - 起動完了
  - 対象デバイス
  - `grab` 成功
  - 設定再読込成功
- `DEBUG`
  - 詳細イベントトレース
  - パススルー/リマップの判定結果

## Testing policy
最低限、以下を用意すること。

### Unit tests
- YAML パース
- 論理検証
- 競合検出
- ルーティング判定
- 未定義イベントのパススルー判定

### Integration tests
- 設定再読込の成功
- 設定再読込の失敗時ロールバック
- 仮想マウス出力の基本動作
- 仮想キーボード出力の基本動作
- systemd 前提のフォアグラウンド起動

### Manual validation
- 指定マウスだけが監視されること
- 物理デバイスが `grab` され、二重入力にならないこと
- リマップ対象ボタンで期待キーが出ること
- 未リマップの移動・ホイール・ボタンが仮想マウス経由で動くこと
- 設定変更が再起動なしで反映されること
- 無効設定でも旧設定が維持されること

## Definition of done
以下を満たした変更のみ完了扱いとする。

- `cargo fmt --all` が通る
- `cargo clippy --all-targets --all-features -- -D warnings` が通る
- `cargo test --all` が通る
- 追加した機能に対応する試験がある
- 受け入れ条件への対応が説明できる
- ログとエラー文が運用で読める品質に達している
- 不要な機能拡張が入っていない

作業前に適切な skill を選ぶこと。

- アーキテクチャ整理、crate 構成、API 設計
  - `rust-remapper-architecture`

  - `linux-input-pipeline`
- YAML、検証、競合、ホットリロード
  - `config-reload-validation`

  - `systemd-daemon-ops`
- 試験、診断、ログ、受け入れ確認
  - `remapper-test-diagnostics`
- Rust 共通規約、lint、error handling
  - `rust-tooling-conventions`

## Request handling style
作業依頼を受けたら、次の順で動くこと。

1. 要件と制約を再確認する
2. 対象モジュールを特定する
3. 最小差分の実装計画を立てる
4. 実装する
5. 試験する
6. 受け入れ条件との対応を報告する

## Explicit non-goals
以下は、明示的に要求されるまで実装しない。

- 複数マウス監視
- ホイールのキーボード変換
- 移動イベントのキーボード変換
- レイヤー切替
- マクロ DSL
- GUI
- Wayland/X11 連携最適化
- クラウド同期
